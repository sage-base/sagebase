# ADR 0015: GCSダンプ方式による開発環境戦略

## Status

Accepted (2026-03-13)

## Context

### 背景

Polibaseの開発環境では、本番データベース（Cloud SQL for PostgreSQL 15）のデータを使って開発・テストを行う必要があります。しかし、Cloud SQLインスタンスを常時稼働させると月額$150程度のコストが発生し、利用頻度が低い開発環境（週1回以下）ではコスト効率が極めて悪くなります。

本番環境はGoogle Cloud Platform上でCloud Run + Cloud SQL構成を採用しており、議事録PDF/テキストはGoogle Cloud Storage（GCS）に `gs://` URI形式で管理されています。この既存のGCSインフラを活用した開発環境戦略が求められていました。

### 課題

1. **コスト問題**: Cloud SQL常時稼働で年間$1,800のコストが発生（開発環境のみ）
2. **利用頻度とのミスマッチ**: 週1回以下の利用頻度に対して、24時間365日の稼働は無駄
3. **データの整合性**: 開発環境でも本番に近いデータで検証したい
4. **起動の手軽さ**: 開発開始時に素早く環境を立ち上げたい

### 検討した代替案

#### Option A: 常時稼働のステージング環境

Cloud SQLインスタンスを常に起動しておく方式。

**利点**: 即座にアクセス可能、データの永続性が保証される、CI/CDでの自動実行に適する
**欠点**: 月額$150、年間$1,800のコスト、利用しない期間も課金が継続

#### Option B: ローカルのみの開発（Docker Compose）

Docker ComposeでローカルにPostgreSQLコンテナを立ち上げ、ローカルのみで完結する方式。

**利点**: クラウドコスト不要、即座に起動可能
**欠点**: 本番データの取得が手動になる、GCS連携のテストが困難、チームメンバー間でのデータ共有が困難

#### Option C: Cloud SQL停止運用（activation-policy）

Cloud SQLの `activation-policy` を使い、未使用時にインスタンスを停止する方式。

**利点**: 起動が比較的高速（1-2分）、データの永続性が保証される
**欠点**: 停止中でも月額$10程度の課金が発生、年間$120-360のコスト

## Decision

**GCSダンプ方式**を開発環境の推奨方式として採用します。

### 方式概要

- **使用時**: Cloud SQLインスタンスを作成し、GCSからデータベースダンプを復元
- **終了時**: GCSにバックアップを保存してCloud SQLインスタンスを削除
- **停止中**: GCSストレージ課金のみ（約$0.02/月）

### 運用フロー

```bash
# 作業開始（初回: 10-15分、2回目以降: 1-2分）
just cloud-up
# または
./scripts/cloud/setup-dev-env.sh

# 作業実施...

# 作業終了（5-10分）
just cloud-down
# または
./scripts/cloud/teardown-dev-env.sh

# バックアップ一覧・状態確認
just cloud-backups
just cloud-status
```

### コスト比較

| 運用方法 | 停止時コスト | 稼働時コスト | 年間コスト（週1日利用） |
|---------|-------------|-------------|---------------------|
| 常時稼働 | $150/月 | $150/月 | $1,800 |
| 停止運用（activation-policy） | $10/月 | $150/月 | $120-360 |
| **GCSダンプ方式** | **$0.02/月** | **日割り** | **$24-96** |

### 適用条件

**推奨ケース**:
- 週1回以下の利用頻度
- 開発・テスト環境
- 個人プロジェクト
- コスト最優先

**非推奨ケース**:
- 毎日使用する環境
- すぐに起動が必要な環境
- CI/CDでの自動実行

## Consequences

### 利点

- **超低コスト**: 停止中はGCS課金$0.02/月のみ、年間約$1,700-2,800削減（週1日利用の場合）
- **データの完全保持**: ダンプファイルがGCSに保存されるため、データ消失リスクが低い
- **バックアップ履歴**: 複数の復元ポイントを保持可能、バックアップ保存先は `gs://sagebase-backups/database-snapshots/`
- **GCS URI統一**: プロジェクト全体で `gs://` URI形式を統一使用しており、既存インフラとの一貫性が保たれる
- **Justfileコマンド統合**: `cloud-up`, `cloud-down`, `cloud-status` コマンドで操作が簡潔

### トレードオフ

- **起動時間**: 初回起動に10-15分かかる（Cloud SQLインスタンス作成 + GCSからの復元）
- **バックアップ・復元の待ち時間**: 作業終了時に5-10分のバックアップ処理が必要
- **完全自動化には不向き**: CI/CDパイプラインでの自動実行には常時稼働環境が適している
- **手順の複雑さ**: 通常のDocker Compose起動と比較して、GCPリソースの操作が加わる
